## Эндпоинты

### Рекомендации мероприятий

Добавьте новый эндпоинт, который возвращает список рекомендованных мероприятий
на основе лайков пользователей.
Рекомендации должны кэшироваться в Redis по принципу Caсhe-Aside (lazy),
чтобы не перестраивать подборку при каждом запросе

> 🔐 Только для авторизованных пользователей.
> Пользователь может запрашивать рекомендации только для себя.

## Спецификация

### Схема данных в Neo4j

**Узлы:**

- `User` — пользователь системы
  - `id` *string* — его идентификатор в MongoDB
- `Event` — мероприятие
  - `id` *string* — его идентификатор в MongoDB
  - `title` *string* — название

**Связи:**

- `(u:User)-[:LIKED]->(e:Event)` — пользователь лайкнул мероприятие

- `LIKED` - связь, показывающая, что пользователь лайкнул то или иное событие

**Алгоритм рекомендаций:**

Для пользователя А:

1. Что лайкнул А? → Мероприятие 1, Мероприятие 2
2. Кто ещё лайкнул Мероприятие 1? → Пользователь Б
3. Кто ещё лайкнул Мероприятие 2? → Пользователь Б, Пользователь В
4. Что лайкнул Пользователь Б? → Мероприятие 3, Мероприятие 4
5. Что лайкнул Пользователь В? → Мероприятие 5
6. Рекомендуем: Мероприятие 3, 4, 5 (которые А ещё не лайкал)

В этой лабораторной под лайками для рекомендаций понимается сам факт
постановки лайка. Если пользователь позже поставил дизлайк, это не должно
удалять связь `LIKED` из рекомендательного графа, потому что дизлайки в этом
алгоритме не учитываются.

### Redis

**Ключ:**

```plaintext
user:{user_id}:recomms
```

`{user_id}` - идентификатор пользователя в MongoDB

```plaintext
user:53e9c0c1a2a3c3d7e6c9c8a1:recomms
```

**Значение:**


## FAQ

**Q: Нужно ли учитывать рейтинг и отзывы в рекомендациях?**  
A: Нет, обойдемся лишь лайками (без дизлайков), так как любое добавление
может сильно поменять алгоритм рекомендаций

**Q: Сколько инстансов Neo4j поднимаем?**  
A: Один. Больше не требуется.

**Q: Нужно ли возвращать `count` в рекомендациях?**  
A: Нет, в рекомендациях только сами рекомендации.

**Q: Что храним в Neo4j?**  
A: Только граф для отбора рекомендаций,
а полные данные мероприятий для ответа берутся из MongoDB.

**Q: Можно не настраивать авторизацию в Neo4j базу?**  
A: Можно. На ваше усмотрение.

**Q: Когда строится граф в Neo4j?**  
A: При создании нового пользователя, события или лайка.
Дизлайки и отзывы не учитываем, так как это усложняет алгоритм.

**Q: Если пользователь сначала лайкнул событие, а потом поставил дизлайк, должен ли этот лайк влиять на рекомендации?**  
A: Да. В этой лабораторной рекомендации строятся только по факту лайков,
а дизлайки для рекомендательного графа не учитываются.

**Q: Нужно ли настраивать инвалидацию рекомендаций в кэше?**  
A: Нет. Достаточно Cache-Aside с TTL,
без обязательной сложной инвалидации чужих рекомендаций.

**Q: Нужно ли сортировать рекомендации?**  
A: Да. Рекомендации должны быть отсортированы по релевантности
— от наиболее популярных (по количеству лайков) к наименее популярным.

**Q: Нужно ли включать в рекомендации события, которые пользователь уже лайкал?**  
A: Нет. Уже лайкнутые пользователем события должны исключаться из рекомендаций.

**Q: Что делать, если в рекомендации попало несколько событий с одинаковым названием?**  
A: Считаем их одним мероприятием и возвращаем только одно событие — ближайшее по дате начала.

**Q: Что делать, если нечего рекомендовать?**  
A: Отдавать пустой список мероприятий. В нашем проекте, это вполне допустимо.
