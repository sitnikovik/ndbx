## Задание

### Создание и обновление сессии

Реализуйте новый endpoint `/session` для управления анонимными сессиями,
который будет создавать сессию при первом визите и обновлять её при повторных визитах.
Будто пользователь сообщает сервису "я здесь", и сервис должен запомнить его на некоторое время.

> Ближайшая аналогия — когда банкомат спрашивает вас *"Вы еще здесь?"*
> если вы долго не совершаете действия, и если вы отвечаете, он продлевает сессию.

Сессия должна создаваться только при первом `POST /session`,
а `GET /health` не должен создавать сессию, но должен возвращать существующую Cookie без обновления TTL.
Если TTL истёк, сессия удаляется и при следующем `POST /session` создаётся **новая** сессия с новым sid.

**Запрос:**

```http
POST http://localhost:8080/session
```

**Ответ при первом визите (создание сессии):**

```http
HTTP/1.1 201 Created
Set-Cookie: X-Session-Id=3f8a2c1d9e4b7f0a5c6d2e8b1a3f9c7d; HttpOnly; Path=/; Max-Age={APP_USER_SESSION_TTL}
Content-Length: 0
```

**Ответ при повторном визите (существующая сессия):**

```http
HTTP/1.1 200 OK
Set-Cookie: X-Session-Id=3f8a2c1d9e4b7f0a5c6d2e8b1a3f9c7d; HttpOnly; Path=/; Max-Age={APP_USER_SESSION_TTL}
Content-Length: 0
```

> ⚠️ Не создавайте и не обновляйте сессию при `GET /health`.
> Это должно происходить **только** при `POST /session`.

**Важно:**

- Первый `/session` создаёт сессию (HTTP 201)
- Повторный `/session` **обновляет TTL** существующей сессии (HTTP 200)
- Cookie устанавливается в обоих случаях

### Health check

Доработайте endpoint `/health` так, чтобы он возвращал существующую сессию в Cookie `X-Session-Id`,
если она была отправлена в запросе, но **НЕ обновлял TTL**.

**Запрос без Cookie (нет сессии):**

```http
GET /health
```

**Ответ:**

```http
HTTP/1.1 200 OK
Content-Type: application/json
{"status":"ok"}
```

**Запрос с Cookie (сессия уже существует):**

```http
GET /health
Cookie: X-Session-Id=a3f8a2c1d9e4b7f0a5c6d2e8b1a3f9c7d
```

**Ответ:**

```http
HTTP/1.1 200 OK
Set-Cookie: X-Session-Id=a3f8a2c1d9e4b7f0a5c6d2e8b1a3f9c7d; HttpOnly; Path=/; Max-Age={APP_USER_SESSION_TTL}
Content-Type: application/json
{"status":"ok"}
```

**Важно:**

- Этот endpoint **НЕ должен** создавать сессию
- Если Cookie **нет** — не устанавливать `Set-Cookie`
- Если Cookie **есть** — вернуть ту же Cookie чтобы браузер не потерял её, но **НЕ обновлять TTL**

### Идентификатор сессии

Идентификатор сессии (далее - *sid*) должен быть криптографически стойким (минимум 128 бит)
и не должен содержать никакой смысловой нагрузки (opaque).

Например, `sid` не должен быть хешем от временной метки,
ip-адреса или User-Agent, и не должен быть UUID.
Рекомендуемый способ генерации `sid` - использование криптографически стойкого генератора
случайных чисел (например, `crypto/rand` в Go, `secrets` в Python) для создания 128-битного токена,
который затем кодируется в **hex** строку.

Идентификатор сессии должен передаваться клиенту в Cookie с именем `X-Session-Id`,
флагом `HttpOnly` и `Max-Age`, соответствующим TTL сессии в Redis.

### Хранение сессий

Сессии должны храниться в Redis с TTL, который определяет весь жизненный цикл сессии.

Ключи в Redis должны иметь формат `sid:{session_id}`,
а данные сессии должны храниться в виде хэш-таблицы (Hash) с полями `created_at` и `updated_at`,
содержащими временные метки создания и последнего обновления сессии в формате RFC3339. Например:

**Ключ**:

```plain
sid:3f8a2c1d9e4b7f0a5c6d2e8b1a3f9c7d
```

**Значение**:

```json
{
  "created_at": "2026-02-14T10:30:00Z",
  "updated_at": "2026-02-14T10:45:00Z"
}
```

> 💡 Создание сессии должно быть атомарным, чтобы избежать гонок при одновременных запросах.
> Используйте опции `NX EX` или аналогичные механизмы в вашем Redis-клиенте вашего ЯП
> для обеспечения атомарности.

### Конфигурация

Добавьте следующие переменные в `.env.local`:

```sh
# Хост, по которому доступно ваше приложение
APP_HOST=localhost
# Порт, на котором оно слушает запросы
APP_PORT=8080
# Время жизни сессии в секундах
APP_USER_SESSION_TTL=60 # 1 минута (ДЛЯ ТЕСТОВ)

# Redis connection
# Хост Redis сервера
REDIS_HOST=your_redis_container_name
# Порт Redis сервера
REDIS_PORT=your_redis_container_port
# Пароль для Redis (оставьте пустым, если пароль не используется)
REDIS_PASSWORD=
# Номер базы данных Redis (по умолчанию - 0)
REDIS_DB=0
```

> ⚠️ Для тестов рекомендуется установить `APP_USER_SESSION_TTL` небольшим, чтобы ускорить проверку истечения сессии.

### Пошаговый flow обработки запроса

#### `GET /health` — чтение без изменений

1. Проверить работоспособность приложения
2. Вернуть **200 OK**
3. **НЕ создавать** сессию
4. **Если Cookie нет** — не устанавливать `Set-Cookie`
5. **Если Cookie есть** — вернуть ту же Cookie с флагом `HttpOnly`, **НЕ обновлять TTL** в Redis

#### `POST /session` — создание и обновление

1. Извлечь `X-Session-Id` из Cookie
2. **Если Cookie нет:**
   - Генерировать новый sid
   - Создать сессию атомарно
   - Установить Cookie с новым sid
   - Вернуть **201 Created**
3. **Если Cookie есть:**
   - Проверить существование сессии в Redis по sid
   - Если сессии нет → создать новую (см. шаг 2)
   - Если сессия существует:
     - Обновить TTL
     - Установить Cookie с тем же sid
     - Вернуть **200 OK**

## FAQ

**Q: Можно ли хранить сессии в строке, а не в хэше?**  
A: Нет. Хэш — обязательное требование.
Он позволяет хранить структурированные данные и атомарно изменять отдельные поля.

**Q: Можно ли по-другому назвать endpoint `/session`?**  
A: Нет. АТ ожидает именно этот endpoint для проверки создания и обновления сессии.

**Q: Нужно ли обновлять TTL при GET `/health`?**  
A: Нет. TTL обновляется **только** при POST `/session`.
GET `/health` — это read-only операция, которая не должна изменять состояние сессии.

**Q: Как генерировать sid?**  
A: Используйте криптографически стойкий генератор (например, `crypto/rand` в Go, `secrets` в Python)
и минимум 128 бит случайности.

**Q: Что делать, если sid уже существует при создании?**  
A: Сгенерировать новый и повторить попытку.
Вероятность коллизии у 128-битного токена ничтожна, но на всякий случай стоит обработать эту ситуацию.

**Q: Можно ли хранить sid в заголовке Authorization?**  
A: Нет. Используйте Cookie.

**Q: Нужно ли валидировать формат sid из Cookie?**  
A: Да. Проверяйте длину и допустимые символы перед использованием в Redis. Это защита от инъекций.

**Q: Можно ли изменить имя Cookie?**  
A: Нет. АТ проверяет точное имя `X-Session-Id`.

**Q: Можно ли изменить формат Redis ключа?**  
A: Нет. АТ проверяет точный формат `sid:{session_id}`.

**Q: Что должно быть в теле ответа `/session`?**  
A: Ничего. Достаточно заголовков и статуса. `Content-Length` должен быть 0.

**Q: Что должен возвращать `/session` при повторном вызове с существующей сессией?**  
A: HTTP 200 OK, обновлённую Cookie и обновлённый TTL в Redis.

**Q: Можно ли не устанавливать `Path=/` в Cookie?**  
A: Технически да (браузер использует текущий путь), но лучше явно указать `Path=/` для совместимости.

**Q: Что будет, если не указать `Max-Age` в Cookie?**  
A: Cookie станет session cookie (удаляется при закрытии браузера),
но сессия в Redis останется до истечения TTL. Это рассинхронизация — плохая практика.

**Q: Нужно ли удалять сессию по истечении TTL?**  
A: Нет. Redis автоматически удаляет ключи по истечении TTL.
Ваша задача — правильно установить TTL при создании и обновлении сессии.

**Q: Можно ли по-другому назвать переменные окружения для Redis?**  
A: Нет. АТ ожидает именно эти переменные.
